# V6 生产可用版本治理总纲

## 1. 文档定位
- 文档状态：`生效`
- 目标：作为 V6 从“当前可联调主链版本”推进到“生产环境可用版本”的唯一宏观治理基线，统一定义交付等级、正式交付边界、模块顺序、开发门禁、联调准入、验收发布与遗留项收口规则。
- 使用边界：本文不替代 `docs/需求方案.md`、`docs/V6项目总控台账.md`、`docs/V6阶段A-流程图状态机与UI原型清单.md`、`docs/V6阶段B-详细设计与原型冻结包.md`、`docs/V6阶段E-验收发布与上线清单.md` 与模块任务包；本文负责把它们串成“生产可用版本”的统一执行口径。
- 唯一原则：凡涉及“是否可真实业务测试、是否可验收、是否可上线”的判断，统一以本文为准，不得仅凭旧版联调通过、局部页面可见、单模块测试通过或演示模式可运行作出结论。

## 2. 当前纠偏结论
- 旧版 `M1 ~ M7`、`M8-01 ~ M8-28` 与阶段 D 首轮联调结果，只能证明旧版范围达到 `可联调`，不能直接推导为 `可真实业务测试`、`可验收` 或 `可上线`。
- 当前缺口不再是单点页面欠账，而是“前置治理基座 + 正式身份体系 + 订单后半状态机 + 发布运维门禁”的系统性收口。
- 因发现旧版阶段判断不足以支撑“生产可用版本”目标，项目已按总控要求回退到阶段 C 重新组织开发；本轮 `G0` 已完成宏观治理文档冻结，后续正式开发入口统一收敛到 `G1 前置治理基座开发`。

## 3. 生产可用版本四级定义

### 3.1 可联调
- 定义：后端、管理后台、小程序在开发或联调条件下可以跑通主链，自动化测试与关键联调通过。
- 允许存在：`demo/proxy/local_api/mock/dev-login/debug_openid` 等开发辅助能力。
- 不代表：真实业务测试可执行、不代表验收通过、不代表可上线。

### 3.2 可真实业务测试
- 定义：在不依赖固定演示角色、脚本伪装身份、手工填写裸 ID 的前提下，真实业务角色可在系统内完成建档、登录、发起、审批、执行、确认、关闭与追溯。
- 必须满足：
  - 组织与公司管理可用；
  - 用户与角色绑定可用；
  - 主数据中心可用；
  - 审批模板与参数配置可用；
  - 管理后台正式认证与小程序正式登录可用；
  - 开发工具能力不再作为业务用户正式入口。

### 3.3 可验收
- 定义：本次版本范围内的业务规则、页面、接口、状态机、权限、审计、报表与异常分支均已通过业务验收与测试验收。
- 必须满足：
  - 无阻断问题；
  - 无未定责问题；
  - 无未处理的一致性争议；
  - 所有降级项、遗留项都有书面承接去向。

### 3.4 可上线
- 定义：在可验收基础上，发布清单、初始化方案、回滚方案、监控告警、审计追踪与观察期机制全部齐备，具备进入生产环境运行条件。
- 必须满足：
  - 发布范围冻结；
  - 回滚方案明确且可执行；
  - 初始化数据与参数装载方案明确；
  - 生产环境禁止开发辅助入口；
  - 业务、产品、研发、测试、运维共同签署 Go/No-Go 结论。

## 4. 本次生产化版本范围与非目标

### 4.1 本次必须完成范围
- `G1 前置治理基座`：组织与公司管理、用户与角色绑定、主数据中心、审批模板与参数配置。
- `G2 正式身份与端收口`：管理后台正式认证、小程序正式登录、会话续期、角色/公司/端权限统一校验。
- `G3 订单后半状态机闭环`：采购订单与销售订单 `执行中/已完成` 状态推进、完成态条件与上下游联动收口。
- `G4 真实业务全链路回归`：合同、订单、资金、库存、关闭、报表、审计统一回归。
- `G5 发布与上线准备`：初始化、发布清单、回滚、观察期与问题升级路径。

### 4.2 本次非目标
- 不扩张当前冻结范围外的新业务域；
- 不把演示能力、美化能力、模板演示页优先级放在治理基座与主链闭环之前；
- 不用“局部可跑”替代“生产可用”；
- 不把治理增强池中的非阻断页面默认拉回本轮并行大改。

## 5. V5 认证继承与正式身份策略

### 5.1 管理后台正式认证继承
- 管理后台正式登录沿用 V5 的 `SUPER_ADMIN + 密码 + TOTP/恢复码` 模型，作为 Web 后台正式认证基线。
- `SUPER_ADMIN` 用于系统治理、账号治理、配置治理与高风险操作准入；普通业务 `ADMIN` 不得替代 `SUPER_ADMIN` 成为公网后台正式主登录入口。
- 管理后台如后续新增企业微信扫码登录，只能作为在本模型基础上的增强方案；未冻结前不得擅自替换当前正式认证基线。

### 5.2 小程序正式认证继承
- 小程序正式登录沿用 V5 的 `wx.login -> code2session/openid -> 账号绑定/激活 -> Bearer 令牌` 模型。
- 未绑定业务角色的微信账号不得直接放行业务页；绑定、激活、启停用与端权限必须由管理后台治理能力统一承接。
- 本地联调令牌与开发者工具快捷入口只能用于开发环境，不得替代正式小程序登录体系。

### 5.3 明确不继承的旧方式
- 不继承 V5 `admin-web` 模板中的 mock 登录、模板二维码页与模板接口契约作为真实业务基线。
- 不继承 V5 开发环境 `wechat_login_mode=mock` 作为生产口径；生产环境必须使用真实微信登录配置。
- 不继续沿用 V6 当前 `demo/proxy` 作为管理后台正式认证方案。

## 6. 管理后台公网安全与正式交付边界
- 管理后台公网访问必须使用 HTTPS、正式域名与证书，禁止裸露开发服务或直接以本地代理模式暴露公网。
- 生产环境下的 `admin-web` 必须通过正式认证体系进入，不得继续使用浏览器侧可构造的开发联调身份、固定身份或演示角色。
- 浏览器端不得直接向生产后端传递 `X-User-Id`、`X-Role-Code`、`X-Company-Id`、`X-Company-Type`、`X-Client-Type`、`X-Auth-Secret` 等开发联调用身份头作为正式认证依据。
- 管理后台必须具备：登录审计、关键操作审计、失败锁定、MFA、正式密钥配置与高风险动作留痕。
- 高风险后台能力（导出、配置、账号治理、模板治理）上线前应同步评估 WAF、限流、IP 白名单、VPN 或零信任接入策略。

## 7. 开发工具能力与正式交付边界
- `demo`、`proxy`、`local_api`、`mock`、`dev-login`、`debug_openid` 统一定义为开发工具能力，不属于正式交付能力。
- 开发工具能力可在本地开发环境保留，但不得作为：
  - 真实业务测试准入依据；
  - 业务验收通过依据；
  - 生产环境运行入口。
- 测试环境若保留调试开关，也不得对业务用户可见；正式测试链路必须尽量逼近生产认证与生产权限边界。
- 所有测试数据、测试附件、测试备注与测试路径必须使用统一可检索标识，并在回归后定向清理。

## 8. 阻断项分级与模块推进顺序

### 8.1 P0：不完成则不能进入真实业务测试
- `G1` 前置治理基座；
- `G2` 正式身份与端收口；
- 演示/联调工具与正式交付边界收口。

### 8.2 P1：不完成则不能通过生产验收
- `G3` 订单后半状态机闭环；
- 完成态、异常分支、权限阻断与中文提示一致性；
- 审计、报表、扫描、重算与业务主链一致。

### 8.3 P2：不完成则不能上线
- `G5` 发布与上线准备；
- 初始化、发布、回滚、监控、观察期、问题升级路径；
- 阶段 E 清单全部通过。

### 8.4 模块推进顺序
1. `G1 前置治理基座`
2. `G2 正式身份与端收口`
3. `G3 订单后半状态机闭环`
4. `G4 真实业务全链路回归`
5. `G5 发布与上线准备`

## 9. 模块开发门禁

### 9.1 开工门禁
- 范围已冻结并与 `docs/需求方案.md`、本总纲、追踪矩阵一致；
- 上下游边界、输入、输出、依赖、状态流转、失败回滚已定义；
- 关键页面/API/模型/测试对象已识别；
- 风险、降级路径与回滚方式已明确。

### 9.2 模块出口门禁
- 实现完成；
- 自动化测试通过；
- 关键主链手工回归通过；
- 文档已同步更新；
- 已完成自检；
- 已完成独立评审；
- 无阻断问题、无高优先级问题、无未处理一致性问题。

## 10. D 到 E 到上线准入口径
- 阶段 D 出口最多代表达到 `可联调` 级别，不得对外表述为 `可真实业务测试` 或 `可上线`。
- 进入真实业务测试前，必须先通过 `G1` 与 `G2`；若仍依赖固定身份、脚本绑定或手工填裸 ID，只能算开发联调模式。
- 进入 UAT 前，必须先通过 `G3`，并确认采购订单、销售订单、合同关闭与报表口径一致。
- 进入阶段 E 前，必须先通过 `G4` 与 `G5`，并满足 `docs/V6阶段E-验收发布与上线清单.md` 的所有准入项。

## 11. 规则-实现-测试-验收追踪机制
- 后续生产化开发统一以 `docs/V6规则-实现-测试-验收追踪矩阵.md` 作为追踪唯一入口。
- 每条关键规则必须能追溯到：规则编号、页面/页面 ID、API、模型/服务、自动化测试、手工验收场景、上线结论。
- 文档冻结但代码未落地的规则，必须明确标注“已冻结、未实现”，不得与已实现状态混淆。
- 任何降级、延后、留坑事项，都必须在追踪矩阵或阶段 E 清单中写明去向，不得口头漂移。

## 12. 当前开发准入判断
- 本轮完成 `G0` 后，项目已具备进入正式功能开发的文档前置条件。
- 当前允许启动的正式功能开发范围仅为 `G1 前置治理基座开发`。
- 当前不允许跳过 `G1` 直接进入 `G2 ~ G5`，也不允许恢复到以 `demo/proxy/local_api` 为主的旧版联调开发节奏。
- 当前推荐的第一个开发入口为：`G1-01 组织与公司管理`。
