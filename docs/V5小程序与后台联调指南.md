# 隽港顺达供应链项目 V5 小程序与后台联调指南

## 1. 文档信息
- 版本：V5
- 更新日期：2026-03-02
- 状态：开发联调基线
- 适用范围：V5 双订单、合同、库存、报表重构联调

## 2. 联调目标
- 打通销售合同、采购合同、销售订单、采购订单、采购入库、库存和报表的完整链路。
- 验证客户、运营、财务、供应商、仓库、业务管理员、SUPER_ADMIN 的角色边界。
- 验证自动生成采购订单、采购合同二次确认、发货指令单模板选择与自动生成、库存冻结释放、异常关闭、角色隔离和审计留痕。

## 3. 终端承载边界
### 3.1 小程序
1. 承接销售订单、采购订单的执行链动作。
2. 承接合同摘要、订单进度、日志、轻量报表的查询与下载。
3. 不承接模板维护、主数据维护、合同新建修改生效作废、采购入库确认、库存调整。

### 3.2 后台
1. 承接协议模板中心、主数据、用户与组织管理。
2. 承接销售合同、采购合同的新建、修改、生效、作废。
3. 承接采购入库确认、库存调整、初始化库存、重报表与审计管理。

## 4. 联调前提
### 4.1 代码与环境
1. 在 `backend/` 目录安装依赖并执行迁移。
2. 小程序联调目录统一使用当前仓库根目录下的 `miniprogram/`；旧版小程序源码已归档到 `archive/2026-03-01-miniprogram-v4-legacy/`，不再参与 V5 联调。
3. 若需同时联调后台管理页面，按当前后台工程实际目录执行启动命令，不再以历史 `admin-web/` 目录为准。
4. 确认附件存储、数据库、日志目录可写。

### 4.2 本地启动
1. 后端
```bash
cd backend
conda activate jgport
pip install -r requirements.txt
alembic upgrade head
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```
2. 管理后台
- 按前端项目实际启动命令执行
3. 微信小程序
- 导入 `miniprogram/`
- `apiBaseUrl` 指向 `http://127.0.0.1:8000/api/v1`

### 4.3 登录模式
- 本地联调默认 `wechat_login_mode=mock`
- 生产联调使用 `official`

### 4.4 库存前提
1. 仓库出库联调前，必须先准备历史初始化库存或已确认采购入库库存。
2. 未完成库存初始化或采购入库确认前，不得直接执行仓库出库联调。

### 4.5 自动化 UAT 约定
1. 自动化 UAT 脚本统一存放在 `tests/uat/`。
2. 全自动 UAT 默认使用 `wechat_login_mode=mock`，不把真实微信 `official` 登录作为每日自动回归前提。
3. 后台管理端自动化、环境清库与编排脚本统一从 `tests/uat/run_full_uat.sh` 串联；小程序自驱 UAT 通过 `ENABLE_MINIPROGRAM_SELF_UAT=1` 显式启用。

## 6. 推荐联调顺序
### 6.1 阶段一：后台基础能力
1. 模板中心
  - 创建模板
  - 设置默认模板
  - 预览 PDF
  - 验证发货指令单模板支持多模板配置
2. 销售合同
  - 新建草稿
  - 生成 PDF
  - 上传盖章合同与保证金回单
  - 提交生效
3. 采购合同
  - 新建草稿
  - 生成 PDF
  - 上传盖章合同与保证金回单
  - 提交生效
  - 校验自动生成采购入库单
4. 库存准备
  - 确认采购入库或录入历史初始化库存
  - 校验库存汇总可用于后续仓库出库链路

### 6.2 阶段二：销售订单主链路
1. 客户提交销售订单
  - 必须关联已生效销售合同
  - 支持历史运输信息一键带入
  - 填写运输资料
  - 提交后运输资料锁定
2. 运营审核
  - 通过
  - 校验销售订单进入财务审批节点并冻结库存
3. 财务审核
  - 上传客户收款凭证
  - 审核通过
  - 校验销售订单进入采购执行中
  - 校验自动生成采购订单

### 6.3 阶段三：采购订单执行链
1. 采购合同自动选择
  - 验证默认排序
  - 验证合同摘要展示
  - 验证二次确认
  - 验证允许手工改选
2. 发货指令单模板选择
  - 验证采购订单提交时必须人工选择模板
  - 验证提交后自动生成发货指令单 PDF
  - 验证采购订单详情内嵌打开链接
3. 向供应商付款
  - 上传付款凭证
4. 供应商审核
  - 查看采购订单详情中的发货指令单 PDF
  - 上传盖章发货指令单
  - 审核通过
  - 校验采购订单进入 `WAREHOUSE_PENDING`
  - 校验销售订单同步进入 `READY_FOR_OUTBOUND`
5. 仓库出库
  - 校验可用库存
  - 上传出库单
  - 实际出库数量不得低于订单数量，且不得超过订单数量的 `110%`
  - 一次出库完成

### 6.4 阶段四：库存与报表
1. 库存调整单
2. 库存流水台账
3. 销售订单报表
4. 采购订单报表
5. 销售合同报表
6. 采购合同报表
7. 库存变动报表
8. 仓库出入库台账

## 7. 关键联调场景
### 7.1 正常完成链路
1. 客户提交销售订单
2. 运营审核通过
3. 校验销售订单进入财务审批节点并冻结库存
4. 财务审核通过并生成采购订单
5. 财务确认采购合同、选择发货指令单模板并上传向供应商付款凭证
6. 校验系统自动生成发货指令单 PDF
7. 供应商审核通过
8. 校验采购订单进入 `WAREHOUSE_PENDING`
9. 校验销售订单同步进入 `READY_FOR_OUTBOUND`
10. 仓库出库完成
11. 校验销售订单、采购订单同步完成

### 7.2 无可用采购合同
1. 财务审核通过
2. 系统自动生成 `PENDING_SUBMIT` 采购订单
3. 后台补录采购合同
4. 提交采购订单继续执行

### 7.3 采购合同自动选择容错
1. 准备两份同油品采购合同
2. 验证系统默认选择待执行量最少的合同
3. 验证页面展示合同号、供应商、待执行数量、预计超执行数量
4. 验证手工改选后可继续提交

### 7.4 历史运输信息带入
1. 准备客户历史订单运输资料
2. 新建销售订单时验证一键带入可用
3. 验证带入后允许客户微调再提交
4. 提交后验证运输资料快照锁定

### 7.5 采购执行失败异常关闭
1. 财务审核通过并已生成采购订单
2. 在采购执行链任意环节由业务管理员触发异常关闭
3. 校验销售订单与采购订单同步变为 `ABNORMAL_CLOSED`
4. 校验冻结库存释放
5. 校验系统不允许原单重开

### 7.6 管理员驳回
1. 在销售订单收款确认前由业务管理员触发驳回
2. 校验销售订单进入 `REJECTED`
3. 若订单已进入财务审批节点，校验冻结库存释放
4. 校验采购订单未生成
5. 校验审计日志记录原因与操作人

### 7.7 合同边界
1. 合同生效后不可编辑
2. 合同关联订单或入库后不可作废
3. 特殊情况仅线下处理

### 7.8 角色可见性
1. 客户仅看销售订单，不看采购订单
2. 供应商、仓库不看收货单位名称
3. 供应商、仓库仅查看运输资料快照，不可修改
4. 发货指令单仅在采购订单详情中查看，销售订单不展示
5. 仓库台账不展示金额

## 9. 联调通过标准
### 9.1 业务链路
- 销售订单与采购订单一一对应
- 财务审核通过后自动生成采购订单
- 自动选采购合同必须二次确认
- 采购订单提交时必须人工选择发货指令单模板并自动生成 PDF
- 采购订单进入 `WAREHOUSE_PENDING` 时销售订单同步进入 `READY_FOR_OUTBOUND`
- 仓库出库完成后销售订单、采购订单、合同执行量、库存同步更新

### 9.2 风险控制
- 销售订单进入财务审批节点时即冻结库存
- 业务管理员执行收款前销售订单驳回或收款后异常关闭时，如已冻结库存则释放
- 采购执行失败时由业务管理员执行双订单同步异常关闭
- 采购订单不提供驳回动作

### 9.3 合规与追溯
- 合同生效后不可修改
- 合同关联合同订单、采购入库、仓库出库后不可作废
- 关键动作均有业务日志与审计日志

### 8.4 角色隔离
- 客户仅可查看销售相关对象
- 供应商仅可查看采购相关对象
- 仓库仅可查看仓库执行相关对象
- 报表字段按角色正确隔离

## 9. 常见问题
### 9.1 看不到采购订单
- 先确认销售订单财务审核是否已通过
- 再确认采购订单是否自动生成成功

### 9.2 采购订单无法提交
- 检查是否已选择采购合同
- 检查是否已完成合同二次确认
- 检查是否已选择发货指令单模板

### 9.3 仓库无法完成出库
- 检查是否已准备历史初始化库存或已确认采购入库库存
- 检查可用库存是否充足
- 检查实际出库数量是否低于订单数量，或是否超过订单数量的 `110%`
- 检查是否已上传出库单
- 检查采购订单是否已进入 `WAREHOUSE_PENDING`

### 9.4 合同无法作废
- 检查合同是否已生效
- 检查是否已关联合同订单、采购入库或仓库出库

### 9.5 报表中心点击"打开报表"无法打开文件
- **根本原因（第一阶段）**：`utils/file.js` 的 `downloadRemoteFile` 函数曾错误指定 `wx.downloadFile` 的 `filePath` 参数，将文件保存到用户目录（`wxfile://usr/...`），导致：
  1. `fs.saveFile` 无法接受用户目录路径（只接受 `wxfile://tmp_...` 临时路径）
  2. 微信基础库 2.12.0 以前的版本，`wx.openDocument` 不支持打开用户目录路径的文件
  3. 所有候选打开路径均为用户目录路径，最终全部失败
- **根本原因（第二阶段）**：后续文件打开链路重构后，部分微信开发者工具环境会返回 `http://tmp/...` 或 `http://usr/...` 形式的伪本地路径；这类路径无法直接交给 `fs.saveFile` 或 `wx.openDocument`，会再次触发“no such file or directory”或“permission denied”。
- **修复方案**：
  1. `downloadRemoteFile` 继续保持“不指定 `filePath`”，优先让微信落临时目录。
  2. 若下载结果不是可直接使用的本地路径，则回退到 `wx.request(responseType=arraybuffer) + fs.writeFile`，先把文件落到真实可打开路径，再执行 `openDocument`。
  3. 仅当上述路径仍不可用时，再尝试额外重试下载，避免把不可用伪路径直接送入打开流程。
- **影响版本**：此问题影响所有 Excel 报表（`.xlsx`）的打开操作，首次修复于 2026-03-04，开发者工具伪路径回归问题于 2026-03-04 再次加固。
