## 2. 迭代8第十四批范围（M8-14 供应商发货确认首批）
- 目标：在不下放资金审核职责的前提下，落地供应商小程序单笔发货确认首批能力，使供应商可对本公司采购订单提交发货确认，并推动状态从`待供应商确认`进入`供应商已确认`。
- 模块边界：本轮处理 `backend + miniprogram + docs` 的供应商发货确认接口、状态推进、确认留痕、页面入口与测试；不进入付款确认、不进入异常关闭、不进入批量确认与批量审核。
- 首批实现边界：
  - 后端新增供应商侧采购订单发货确认接口，提交范围限定为当前供应商公司名下采购订单。
  - 状态推进仅允许 `待供应商确认` -> `供应商已确认`，不得跳级推进。
  - 小程序在 `MINI-SUPPLIER-PO-01` 中新增单笔确认按钮、确认结果提示与状态刷新。
  - 页面允许填写简短确认说明，但不承载资金结算、异常驳回、批量确认等复杂动作。
  - 若订单已有附件摘要，页面应在确认前展示“已留痕附件数”；若无附件，仅提示建议补齐，不强制阻断确认。
- 本轮不实现：
  - 供应商付款确认
  - 采购订单异常关闭、驳回、改单
  - 批量发货确认、批量审核
  - 与仓储出库单的自动联动推进
  - 发货确认后的自动付款校验触发

## 2.2 M8-14 实施约束
- 发货确认接口必须按 `supplier + supplier_company + miniprogram` 做服务端权限校验，不得仅依赖前端按钮显隐。
- 发货确认必须校验采购订单当前状态，仅允许在 `待供应商确认` 状态下提交；其余状态重复确认需明确阻断。
- 确认说明、测试数据或演示备注必须使用中文，并在测试环境中带统一可检索标识，如 `CODEX-TEST-`。
- 页面文案必须明确“发货确认不等于付款确认，也不等于仓储已执行完成”，避免用户误解状态含义。
- 若确认成功后状态刷新失败，前端需给出中文提示并允许用户手动重试详情加载。

## 3. 上下游边界
- 上游输入：`M8-12` 已开放的供应商采购订单详情页、`M8-13` 已具备的附件摘要能力、采购订单状态机。
- 下游输出：供应商可提交单笔发货确认，采购订单进入`供应商已确认`，后续由付款校验和执行链继续推进。
- 外部依赖：采购订单详情接口、审计日志能力、小程序登录会话、演示模式 mock。
- 失败回滚：若权限、状态流转或审计字段不满足要求，则回滚本轮确认接口与页面入口，恢复只读+附件回看态。

## 4. 业务价值与优先级判断
- 业务可执行性：高。供应商确认“已完成发货准备”是采购协同中的真实高频动作。
- 运营效率与成本：中高。可减少线下口头确认与截图确认，提升订单推进透明度。
- 运输与仓储安全风险：中。仅推进到“供应商已确认”，不直接触发仓储放行，风险可控。
- 合规性、可审计性与可追溯性：高。确认时间、确认人、确认前状态都可作为审计链路证据。
- 实施复杂度与迭代优先级：中。基于现有详情页和附件链路扩展，复杂度低于付款确认。
