# V6 实施统筹与模块拆分清单（不开发代码）

## 1. 统筹决策
- 结论：采用“三段式推进”，顺序固定为：
  1. 流程图 / 状态机 / UI 原型清单
  2. 详细设计与原型冻结
  3. 模块拆分到子模块并按迭代开发
- 原因：先冻结业务流与交互边界，再冻结数据与接口，最后进入开发，可显著降低返工与上下文遗失风险。

## 2. 阶段与产出

### 阶段A：流程图 / 状态机 / UI 原型清单
- 目标：冻结业务流、状态流、端能力边界。
- 产出：
  - 合同、订单、收付款单、出入库单全链路流程图。
  - 核心状态机：合同状态、订单状态、单据生效状态、手工关闭状态。
  - UI 原型清单：页面目录、关键交互、异常态、空态、审批态。
  - 端能力矩阵：管理后台 vs 小程序页面与权限边界。
- 通过门槛：业务方、财务、仓储三方评审通过。

### 阶段B：详细设计与原型冻结
- 目标：冻结实现口径，形成可开发规格。
- 产出：
  - 数据模型设计：合同、订单、收付款单、出入库单、上下游关系模型。
  - 触发矩阵：自动生成与手工补录触发事件、幂等策略、失败补偿。
  - 接口契约：字段、校验、错误码、审计字段、版本策略。
  - 原型冻结包：页面组件规范、字段口径、交互动作字典。
- 通过门槛：后端、前端、测试联合评审通过。

### 阶段C：模块拆分与迭代开发
- 目标：单模块闭环，逐步联调，不跨模块大并发开发。
- 产出：每迭代模块的开发、测试、文档、回归记录。
- 通过门槛：模块自测通过 + 文档更新 + 关键回归通过。

## 3. 模块拆分（一级）

### M1 基础能力模块
- 子模块：组织与角色、权限与菜单、审批流引擎、审计日志。
- 依赖：无。

### M2 合同域模块
- 子模块：采购合同、销售合同、合同状态机、合同阈值规则。
- 依赖：M1。

### M3 订单域模块
- 子模块：销售订单、采购订单、销售单转采购单联动、订单状态机。
- 依赖：M1、M2。

### M4 资金单据模块
- 子模块：收款单、付款单、保证金单据类型、零金额例外规则。
- 依赖：M2、M3。

### M5 库存执行模块
- 子模块：入库单、出库单、仓库执行回执、手工补录出库。
- 依赖：M2、M3。

### M6 履约闭环与关闭模块
- 子模块：数量履约计算、金额闭环计算、自动关闭、手工关闭处理顺序。
- 依赖：M2、M3、M4、M5。

### M7 报表与看板模块
- 子模块：管理后台仪表盘、业务看板、多维报表、小程序轻量报表。
- 依赖：M2~M6。

### M8 端能力落地模块
- 子模块：管理后台全业务页面、小程序轻量业务页面、端权限校验。
- 依赖：M1~M7。

## 4. 子模块迭代建议（示例）
1. 迭代1：M1-组织角色 + M1-审计基础
2. 迭代2：M2-采购合同 + M2-销售合同
3. 迭代3：M3-销售订单 + 销售转采购联动
4. 迭代4：M4-收付款单 + 保证金单据类型
5. 迭代5：M5-入库单 + 出库单（含手工补录）
6. 迭代6：M6-闭环计算 + 手工关闭
7. 迭代7：M7-仪表盘/看板 + 报表SLA
8. 迭代8：M8-小程序轻量化与端边界回归

## 5. 管理后台与小程序功能划分（决策）

### 管理后台（全量业务）
- 合同、订单、收付款、出入库全流程操作。
- 主数据维护、阈值配置、审批流配置、手工关闭、批量处理。
- 仪表盘、业务看板、多维报表、审计查询与导出。
- 后台登录角色限定：运营、财务、管理员（均归属运营商公司）。

### 小程序（移动适配业务）
- 待办处理、订单发起与进度查询。
- 仓库执行回执、简化出入库确认。
- 轻量报表查看、异常提醒与消息通知。
- 不开放复杂主数据配置、复杂财务对账、批量改单与批量关闭。
- 客户、供应商、仓库角色用户仅通过小程序处理业务。
