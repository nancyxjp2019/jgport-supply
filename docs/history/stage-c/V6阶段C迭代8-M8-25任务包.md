## 2. 迭代8第二十五批范围（M8-25 多维报表钻取首批）
- 目标：在不改动 `metric_version=v1` 汇总口径的前提下，补齐“多维报表汇总行 -> 资金单据明细页”的首批钻取闭环，提升财务与运营对异常资金状态的穿透效率。
- 模块边界：本轮处理 `admin-web多维报表页 + 资金处理台/退款核销台筛选承接 + backend资金列表筛选补齐 + docs`；不进入小程序、不补图形钻取、不做报表口径重算。
- 首批实现边界：
  - 多维报表在 `group_by=doc_status` 时，支持从汇总行分别钻取到收款单、付款单明细列表。
  - 多维报表在 `group_by=refund_status` 时，支持从汇总行分别钻取到收款单、付款单退款/核销处理列表。
  - 资金页与退款核销页需承接路由查询参数并自动回填筛选条件，保证跨页后仍可追溯来源维度值。
  - 后端资金列表接口首批补齐 `refund_status` 筛选，避免仅靠前端本地过滤导致列表截断与口径偏差。
  - 演示模式补齐钻取链路 mock 与单测，后端补齐列表筛选回归测试。
- 本轮不实现：
  - `contract_direction` 维度的真实单据明细钻取与合同/订单联动跳转
  - 报表图形钻取、趋势联动、跨页多条件联动编排
  - 口径重算任务、报表重放、历史版本回算与自动补偿链路

## 2.2 M8-25 实施约束
- 钻取入口、来源提示、筛选标签与错误文案统一中文显示，不向终端透传英文异常原文。
- 仅允许对 `doc_status`、`refund_status` 两类维度行开放钻取；`contract_direction` 维度继续只做汇总回看，不在本轮伪造明细跳转。
- 多维报表钻取不得改变既有汇总口径，不得在页面侧重算金额与数量；明细页仅承接筛选条件并回看资金单据。
- 资金页、退款核销页继续沿用 `M8-22/M8-23` 权限边界：运营只读，财务/管理员可操作，不因报表钻取放宽服务端门禁。
- 路由 query 必须可追溯来源页面与维度值，便于审计回看和手工复核。
