## 2. 迭代8第十三批范围（M8-13 供应商附件回传首批）
- 目标：在不下放资金确认职责的前提下，落地供应商小程序附件回传首批能力，支撑供应商上传盖章发货指令单或业务回单类附件，并让采购订单详情可回看附件摘要。
- 模块边界：本轮处理 `backend + miniprogram + docs` 的供应商附件上传、附件元数据留痕、采购订单详情附件摘要与基础测试；不进入付款确认、不进入异常关闭、不进入批量上传与批量审核。
- 首批实现边界：
  - 后端新增供应商侧采购订单附件上传与查询接口，查询/上传范围限定为当前供应商公司。
  - 复用现有 `doc_attachments` 机制，补充采购订单附件业务标签与查询封装，保证审计可追溯。
  - 小程序在 `MINI-SUPPLIER-PO-01` 中将“附件入口预留”升级为真实上传入口与已上传摘要。
  - 页面仅开放单笔采购订单维度的附件上传与回看，不做跨订单聚合列表。
  - 演示模式与本地联调模式都需要提供可验证的附件摘要反馈，但不得伪造“付款已确认”等资金状态。
- 本轮不实现：
  - 供应商付款确认
  - 发货确认提交动作
  - 采购订单异常关闭、驳回、改单
  - 多附件批量拖拽、批量删除、批量下载
  - OSS 直传优化与真机域名联调细节

## 2.2 M8-13 实施约束
- 供应商上传接口必须按 `supplier + supplier_company + miniprogram` 做服务端权限校验，不得仅依赖前端页面入口控制。
- 附件必须强绑定具体采购订单，且业务标签首批固定在“盖章发货指令单/供应商回单”范围内，禁止透传任意标签。
- 上传生成的测试数据或测试路径必须带统一标识，如 `CODEX-TEST-` 或 `AUTO-TEST-`，回归后需定向清理。
- 页面文案必须全中文，并明确提示“当前仅开放附件回传首批，不代表已完成付款确认或发货审核”。
- 若对象存储或文件上传能力未最终冻结，需先以受控路径方案实现元数据闭环，再在后续迭代切换存储介质，不得跳过审计字段。

## 3. 上下游边界
- 上游输入：`M8-12` 已开放的供应商采购订单详情页、现有 `doc_attachments` 表、资金单据附件处理能力。
- 下游输出：供应商可上传并回看采购订单附件，运营/财务后续可沿采购订单追溯附件留痕。
- 外部依赖：文件上传策略、采购订单详情接口、供应商身份绑定、审计日志能力。
- 失败回滚：若附件归属、权限范围或审计字段不满足要求，则回滚本轮上传接口与页面入口，恢复附件占位态。

## 4. 业务价值与优先级判断
- 业务可执行性：高。供应商上传盖章单据符合实际油品采购协同习惯，便于运营追单。
- 运营效率与成本：中高。先补附件留痕可减少微信/线下散落传图，降低人工对账成本。
- 运输与仓储安全风险：中。只开放回传不开放执行确认，避免将现场放行责任过早转移到小程序。
- 合规性、可审计性与可追溯性：高。附件回传是后续资金确认、发货审核的关键证据链。
- 实施复杂度与迭代优先级：中。可复用现有附件模型与资金附件校验逻辑，风险低于直接开放付款确认。
