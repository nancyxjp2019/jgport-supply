### 2.2 M4第一批已冻结结果
- 自动生成的合同保证金收付款单已落为 `草稿`，金额初始化为 `0.00`，待后续财务确认流程补齐。
- 销售订单衍生的收付款单已根据任务金额生成 `草稿` 单据。
- 销售订单衍生采购订单 `付款金额=0` 场景，付款单草稿已写入例外标识与中文提示文案，保留后续补录能力。
- 手工补录付款单已强制绑定 `采购合同 + 采购订单`，且两者必须互相匹配。
- 手工补录收款单已强制绑定 `销售合同 + 销售订单`，且两者必须互相匹配。
- `doc_relations` 的基础追溯已覆盖：
  - 合同 -> 保证金收付款单
  - 销售订单 -> 收款单
  - 采购订单 -> 付款单
  - 销售订单 -> 采购订单

### 2.3 M4第二批实施边界
- 新增 `POST /api/v1/receipt-docs/{id}/confirm`、`POST /api/v1/payment-docs/{id}/confirm`。
- 非0金额确认时必须上传至少一个凭证路径，并落库到 `doc_attachments`。
- 已转 `待补录金额` 的单据，允许继续调用原确认接口补录金额与凭证后再次确认。
- `付款金额=0` 且命中规则11时无条件放行，直接确认为 `已确认`，并保留统一文案 `例外放行（需后补付款单）`。
- `收款金额=0` 或非规则11的 `付款金额=0` 时，按规则14执行阈值校验：
  - 通过：转 `已确认`，记录免凭证原因。
  - 不通过：转 `待补录金额`，阻断本次确认。
- 规则14实现口径冻结为：
  - 仅适用于“订单衍生的普通收款单/付款单”。
  - 保证金对应数量 = `已确认/已核销且未退款的保证金净额 / 当前订单油品对应合同单价`
  - 待执行数量：
    - 销售方向：`contract_item.qty_signed - contract_item.qty_out_acc`
    - 采购方向：`contract_item.qty_signed - contract_item.qty_in_acc`
- 本批次仍不实现：
  - 退款/核销
  - 收付款单驳回/待审核独立接口
  - 真实文件上传，仅接收并落库存量化后的凭证路径
