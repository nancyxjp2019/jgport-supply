# V6 阶段C后续能力路线图

## 1. 结论先行
- 当前仓库并不存在“供应商功能明显多于运营/财务核心业务后端”的事实；`M3~M7` 已完成订单、资金、库存、合同关闭、报表等主链规则与接口。
- 本文档最初用于解释阶段C迭代8早期“供应商任务包更多、后台页面任务包更少”的原因；相关后台页面能力已在 `M8-16 ~ M8-23` 按冻结顺序补齐。
- `M8-24` 导出编排任务中心首批、`M8-25 多维报表钻取首批`、`M8-26 每日闭环扫描与履约滞留扫描首批` 已完成；当前唯一主线已冻结为 `M8-27 汇总报表口径重算首批`。

- 以上能力的价值在于“可看见主链摘要”，但尚不足以替代真正的后台处理台。

### 3.3 文档颗粒度结论
- 供应商协同与运营/财务后台能力现已都拆到页面级任务包，早期“供应商文档更多”的观感差异已基本收敛。
- 后续 `M8-24+` 应继续沿用页面/能力级任务包颗粒度，避免再次出现“大包规则 + 小包页面”混杂带来的认知偏差。
- `docs/README.md` 已将现行基线、历史归档与规划文档分层，后续新增文档按该结构维护。

## 4. 历史原因判断
- 第一，`M8` 的主线是端能力落地，而供应商此前几乎没有真实业务页，因此被拆成连续小任务包逐步补齐。
- 第二，运营/财务的主业务规则已在 `M3~M7` 先完成后端主链，后续后台页面迁移没有同步按同等粒度拆包。
- 第三，本文档形成时 `admin-web` 仍主要停留在登录、仪表盘、业务看板等基础页面；该差距已在 `M8-16 ~ M8-23` 收敛。
