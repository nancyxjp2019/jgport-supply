## 2. 迭代8第十五批范围（M8-15 供应商付款校验结果回看首批）
- 目标：在不下放资金审核职责的前提下，落地供应商小程序对采购订单付款校验结果的首批只读回看能力，让供应商能理解“已确认发货后，当前是否仍在待付款校验、是否已可继续执行、是否命中零金额例外”。
- 模块边界：本轮处理 `backend + miniprogram + docs` 的付款校验结果只读口径、采购订单详情补充字段、中文提示文案与测试；不进入供应商侧付款确认、不进入异常关闭、不进入批量处理与资金审批操作。
- 首批实现边界：
  - 后端补齐供应商采购订单详情中的付款校验结果展示字段，范围限定为当前供应商公司名下采购订单。
  - 小程序在 `MINI-SUPPLIER-PO-01` 中新增“付款校验结果”只读卡片或提示区，展示当前状态、中文说明与例外放行提示。
  - 首批允许展示 `待付款校验`、`可继续执行` 等付款校验结果说明，并对`执行中`补充“已完成付款校验并进入后续执行”的承接提示，但不展示后台内部审批人、内部待办编号等运营字段。
  - 对 `zero_pay_exception_flag=true` 的采购订单，页面需明确区分“例外放行后可继续执行”与“仍需后补付款单”的提示。
  - 演示模式与本地联调模式都需提供可验证反馈，但不得伪造“已付款”或“财务已完成终审”的资金结论。
- 本轮不实现：
  - 供应商付款确认
  - 供应商上传付款凭证
  - 采购订单异常关闭、驳回、改单
  - 跨订单批量筛选、批量确认、批量审核
  - 真实消息推送或订阅通知

## 2.2 M8-15 实施约束
- 供应商侧仅允许“看结果”，不得新增任何会修改付款状态、收付款单据或资金审计结论的操作按钮。
- 结果回看必须按 `supplier + supplier_company + miniprogram` 做服务端权限隔离，不得透出其他公司采购订单的付款链路信息。
- 页面文案必须明确“付款校验结果回看不等于供应商付款确认”，避免把后台资金职责错误下放到供应商侧。
- 对 `zero_pay_exception_flag=true` 的场景必须给出中文例外提示，但不得误写为“无需后续补录”或“已完成付款”。
- 若当前后端无法提供完整资金结果字段，首批可先以冻结状态映射和说明文案实现只读回看，不得伪造真实付款流水。

## 3. 上下游边界
- 上游输入：`M8-14` 已开放的供应商发货确认结果、`M4` 资金单据规则、采购订单状态机、零金额例外标记。
- 下游输出：供应商可在采购订单详情页理解付款校验是否推进、当前是否可继续执行，以及是否需要等待后续补录。
- 外部依赖：采购订单详情接口、资金规则映射、供应商身份绑定、小程序登录会话、演示模式 mock。
- 失败回滚：若结果口径、权限范围或中文说明不满足冻结边界，则回滚本轮结果字段与页面卡片，恢复到仅展示状态名的能力边界。

## 4. 业务价值与优先级判断
- 业务可执行性：高。供应商在发货确认后最关心是否卡在付款校验，补齐只读反馈可降低线下追问频率。
- 运营效率与成本：中高。明确状态解释后，可减少运营/财务重复答疑与截图通知成本。
- 运输与仓储安全风险：中低。首批只开放回看，不开放资金操作，不会直接放大资金与放行风险。
- 合规性、可审计性与可追溯性：高。通过统一口径展示付款校验结果，可让供应商理解状态，但不越过后台审计边界。
- 实施复杂度与迭代优先级：中。可优先基于现有状态机和例外标记做只读展示，复杂度低于开放真实付款确认。
