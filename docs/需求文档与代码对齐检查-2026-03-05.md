# 需求文档与代码对齐检查报告（2026-03-05）

## 1. 检查范围
- 文档：`docs/需求方案.md`、`docs/V5状态机定义文档.md`、`backend/README.md`
- 代码：`backend/app/models`、`backend/app/schemas`、`backend/app/services`、`backend/app/api/routers`
- 检查目标：确认当前“生效”口径与代码实现一致，识别并修正可确认的不一致项。

## 2. 已确认并修正的不一致项

### 2.1 订单数量精度口径
- 问题：`docs/需求方案.md` 中“数量(吨)最多2位小数”与代码校验“最多4位小数”不一致。
- 代码依据：`backend/app/schemas/v5_sales_order.py`（`qty_ton` 校验为最多4位小数）。
- 已修正：文档改为“最多4位小数”。

### 2.2 采购订单提交流转口径
- 问题：状态机文档仍描述 `PENDING_SUBMIT -> SUPPLIER_PAYMENT_PENDING -> SUPPLIER_REVIEW_PENDING` 两段流转；但当前代码为提交时同步上传付款凭证，直接进入 `SUPPLIER_REVIEW_PENDING`。
- 代码依据：`backend/app/services/v5_purchase_order_service.py` 中 `submit_purchase_order` 直接赋值 `SUPPLIER_REVIEW_PENDING`。
- 已修正：
  - `docs/V5状态机定义文档.md` 更新状态图与流转表；
  - 明确 `SUPPLIER_PAYMENT_PENDING` 仅保留历史兼容。

### 2.3 合同作废边界口径
- 问题：状态机文档曾写“`EFFECTIVE` 可作废且无关联单据”；当前代码仅允许 `DRAFT`、`PENDING_EFFECTIVE` 作废。
- 代码依据：`backend/app/services/v5_contract_service.py` 的 `_VOIDABLE_CONTRACT_STATUSES`。
- 已修正：`docs/V5状态机定义文档.md` 改为仅草稿/待生效可作废。

### 2.4 采购入库作废能力口径
- 问题：状态机文档包含采购入库“作废”动作；当前代码未提供作废接口与服务实现。
- 代码依据：
  - `backend/app/models/v5_domain.py` 定义了 `PurchaseStockInStatus.VOIDED`；
  - `backend/app/api/routers/v5_inventory.py` 仅有 `confirm`，无 `void`。
- 已修正：状态机文档改为“`VOIDED` 为预留状态，当前版本未开放作废动作”。

### 2.5 报表状态过滤描述
- 问题：文档描述排除 `CANCELLED`、`REJECTED`、`ABNORMAL_CLOSED`；当前代码不存在 `CANCELLED` 枚举且报表查询未做状态排除。
- 代码依据：`backend/app/services/v5_report_service.py` 订单报表查询无状态排除条件。
- 已修正：文档改为“按日期范围导出当前查询范围订单；当前实现不额外排除特定状态”。

### 2.6 后端接口清单
- 问题：`backend/README.md` 仍列出 `PATCH /api/v1/purchase-orders/{id}/supplier-payment`，但路由未注册。
- 代码依据：`backend/app/api/routers/v5_purchase_orders.py` 无该接口。
- 已修正：移除该接口条目。

## 3. 对齐结论
- 当前核心业务链路（销售订单、采购订单、合同、采购入库确认、库存、终止动作）与文档已完成本轮对齐。
- 已确认的差异均已落地修正为“文档对齐当前代码实现口径”。

## 4. 残余风险与后续建议
- `SUPPLIER_PAYMENT_PENDING` 仍保留在枚举与部分兼容逻辑中，建议在新项目中明确“保留兼容”或“彻底移除”。
- 采购入库 `VOIDED` 状态已预留但未开放动作，建议在新项目立项阶段明确是否实现该闭环。
- 报表是否排除终止态需在新项目中固定为单一口径，避免经营看板与审计导出口径冲突。
