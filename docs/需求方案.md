# V6 需求方案

## 业务规则清单

### 1. V6 重建主策略
- 状态：`生效`
- 规则：V6 采用独立工程重建方式，不在 V5 目录上直接迭代开发。
- 说明：所有新需求、重构任务、测试与验收均以 V6 目录为唯一交付对象。

### 2. V5 归档策略
- 状态：`生效`
- 规则：V5 代码、文档、脚本统一归档到 `archive/v5/`，仅用于历史对照与追溯。
- 说明：禁止将 V6 新增代码直接提交到 `archive/v5/`。

### 3. V6 生产部署数据策略
- 状态：`生效`
- 规则：V6 在生产环境采用全新部署与全新数据库，不执行 V5 老数据迁移。
- 说明：V5 数据仅用于历史查询与审计对照，不进入 V6 交易链路。

### 4. 业务域迁移顺序
- 状态：`规划中`
- 规则：按“用户与权限 -> 合同与订单 -> 收付款与出入库 -> 报表与审计”顺序分阶段迁移。
- 说明：每阶段需提供可回归测试口径与审计链路说明。

### 5. 合规与审计连续性
- 状态：`生效`
- 规则：V6 重建期间必须保留终止类动作的原因留痕、二次确认与审计日志。
- 说明：不得因前端或模型改造削弱可追溯性。

### 6. 金额单据中台化
- 状态：`生效`
- 规则：合同、订单产生的所有金额流转统一由收款单、付款单承载与跟踪。
- 说明：系统自动生成为主，手工录入时必须绑定合同与业务方向（采购/销售）。

### 7. 数量单据中台化
- 状态：`生效`
- 规则：合同、订单产生的所有数量流转统一由入库单、出库单承载与跟踪。
- 说明：系统自动生成为主，手工录入时必须绑定合同、油品与仓库。

### 8. 采购合同入库流程
- 状态：`生效`
- 规则：采购合同生效后系统自动生成入库单草稿，仓库需手工录入实际入库数量并提交后生效。
- 说明：未提交生效前不计入合同履约数量与库存结余。

### 9. 销售合同出库流程
- 状态：`生效`
- 规则：销售合同对应出库量由两类单据构成：仓库正常流程出库单 + 手工补录出库单。
- 说明：两类出库单均必须手工绑定销售合同，且共同参与合同履约计算。

### 10. 双阈值配置模型
- 状态：`生效`
- 规则：阈值拆分为两个业务场景独立参数。
- 说明：
  - 保证金放行阈值（默认 `1.05`）：用于判断订单是否允许零金额且免凭证。
  - 合同超量履约阈值（默认 `1.05`，即 5%）：用于限制累计实际履约数量上限。
  - 约束：保证金放行阈值不得大于合同超量履约阈值，避免口径冲突。

### 11. 零金额与免凭证例外
- 状态：`生效`
- 规则：当合同待执行数量（按油品）小于等于保证金对应数量乘以保证金放行阈值时，订单可填写收/付款金额为 0，且可免上传收/付款凭证。
- 说明：该场景仍需生成收付款单并记录“免凭证原因、操作人、操作时间”。

### 12. 订单定价与实收实付
- 状态：`生效`
- 规则：订单单价来源于对应合同单价，实收实付金额来源于财务确认的收付款单金额。
- 说明：系统自动生成收付款单，附件归属收付款单而非订单。

### 13. 合同数量履约状态
- 状态：`生效`
- 规则：当合同累计实际履约数量达到合同数量时，合同进入“数量履约完成”状态。
- 说明：数量履约完成不等于合同最终关闭。

### 14. 合同金额闭环判定（采购与销售分开）
- 状态：`生效`
- 规则：合同最终关闭前必须满足对应方向的金额闭环公式。
- 说明：
  - 销售合同：`累计实际出库数量 × 合同单价 = 销售保证金 + 累计收款净额`
  - 采购合同：`累计实际入库数量 × 合同单价 = 采购保证金 + 累计付款净额`
  - 上述口径均按合同方向独立核算，不得将收款与付款相加混算。

### 15. 合同最终关闭与手工关闭
- 状态：`生效`
- 规则：合同自动最终关闭需同时满足“数量履约完成 + 金额闭环”；不满足时仅允许授权用户手工关闭。
- 说明：手工关闭必须“原因必填 + 二次确认 + 审计留痕”。

### 16. 手工关闭后的业务处理
- 状态：`生效`
- 规则：合同手工关闭后，关联单据全部转入手工关闭链路，剩余数量和剩余金额停止跟踪。
- 说明：
  - 关闭后单据不得被新单据引用。
  - 已过账库存流水不回滚。
  - 未完成的出入库预占/预留释放。
  - 未完成单据转“手工关闭/作废”并记录原因。

### 17. 合同履约数量上限
- 状态：`生效`
- 规则：合同累计实际履约数量不得超过合同数量乘以合同超量履约阈值（默认 1.05）。
- 说明：超限时禁止审批通过或禁止单据生效。

### 18. 单据上下游可追溯
- 状态：`生效`
- 规则：合同、订单、收付款单、出入库单均必须记录上下游关联关系。
- 说明：关联关系需支持一对多和多对多，满足审计穿透与报表追溯需求。

### 19. 报表分层策略
- 状态：`规划中`
- 规则：小程序与管理后台采用差异化报表策略。
- 说明：
  - 小程序：提供轻量实时汇总报表（查询简单、字段收敛）。
  - 管理后台：提供多维管理报表（合同、订单、油品、客户、仓库、时间、异常等）。

### 20. 报表生成机制
- 状态：`规划中`
- 规则：采用“事件触发增量更新 + 定时全量校准”的混合机制，不采用纯同步触发式重算。
- 说明：减少高并发下报表阻塞风险，提升可审计性与可追溯性。
