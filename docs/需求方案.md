# V6 需求方案

## 业务规则清单

### 1. V6 重建主策略
- 状态：`生效`
- 规则：V6 采用独立工程重建方式，不在 V5 目录上直接迭代开发。
- 说明：所有新需求、重构任务、测试与验收均以 V6 目录为唯一交付对象。

### 2. V5 归档策略
- 状态：`生效`
- 规则：V5 代码、文档、脚本统一归档到 `archive/v5/`，仅用于历史对照与追溯。
- 说明：禁止将 V6 新增代码直接提交到 `archive/v5/`。

### 3. V6 生产部署数据策略
- 状态：`生效`
- 规则：V6 在生产环境采用全新部署与全新数据库，不执行 V5 老数据迁移。
- 说明：V5 数据仅用于历史查询与审计对照，不进入 V6 交易链路。

### 4. 业务域迁移顺序
- 状态：`规划中`
- 规则：按“用户与权限 -> 合同与订单 -> 收付款与出入库 -> 报表与审计”顺序分阶段迁移。
- 说明：每阶段需提供可回归测试口径与审计链路说明。

### 5. 合规与审计连续性
- 状态：`生效`
- 规则：V6 重建期间必须保留终止类动作的原因留痕、二次确认与审计日志。
- 说明：不得因前端或模型改造削弱可追溯性。

### 6. 金额单据中台化
- 状态：`生效`
- 规则：合同、订单产生的所有金额流转统一由收款单、付款单承载与跟踪。
- 说明：
  - 保证金在系统中定义为收款单/付款单的业务类型，不单独作为额外金额来源。
  - 系统自动生成为主，手工录入时必须绑定合同与业务方向（采购/销售）。

### 7. 数量单据中台化
- 状态：`生效`
- 规则：合同、订单产生的所有数量流转统一由入库单、出库单承载与跟踪。
- 说明：
  - 系统自动生成为主，手工录入时必须绑定合同、油品与仓库。
  - 出入库单必须显式记录油品、仓库、来源类型与可追溯业务键，禁止仅靠上游单据隐式推断。

### 8. 采购合同生效触发规则
- 状态：`生效`
- 规则：采购合同生效时系统自动生成两类单据：付款单（保证金）与入库单（草稿）。
- 说明：
  - 若采购合同存在多个油品明细，系统按合同油品明细逐条生成入库单草稿。
  - 入库单需仓库手工录入实际入库数量并提交后才生效。
  - 未生效入库单不计入合同履约数量与库存结余。

### 9. 销售合同生效触发规则
- 状态：`生效`
- 规则：销售合同生效时系统自动生成收款单（保证金）。
- 说明：保证金收款单纳入合同资金台账与闭环核算。

### 10. 销售订单财务审批触发规则
- 状态：`生效`
- 规则：销售订单进入财务审批通过后，系统自动生成：采购订单、付款单（采购订单实付金额）、收款单（销售订单实收金额）。
- 说明：
  - 自动生成单据必须保留来源单据编号与上下游关联关系。
  - 订单状态机冻结为细粒度并作为唯一业务判定标准，不允许再以“审批中”等聚合状态作为接口前置条件。
  - 销售订单标准状态：`草稿 -> 待运营审批 -> 待财务审批 -> （驳回或已衍生采购订单） -> 执行中 -> 已完成`。
  - 采购订单标准状态：`已创建 -> 待供应商确认 -> 供应商已确认 -> 待付款校验 -> 可继续执行 -> 执行中 -> 已完成`。
  - 销售订单关闭（`已完成`）条件：
    - 关联出库单（系统+手工）在防重后累计实际出库量 `>=` 订单数量，且未触发合同超量履约阈值阻断；
    - 关联采购订单状态为`已完成`；
    - 关联收款单状态为`已确认/已核销`，或满足规则14的0金额例外并留痕。
  - 采购订单关闭（`已完成`）条件：
    - 供应商确认与仓库执行链路完成；
    - 关联付款单状态为`已确认/已核销`，或满足规则11的0付款例外并留痕；
    - 与销售订单的上下游关系完整且可追溯。

### 11. 销售订单衍生采购订单零付款规则
- 状态：`生效`
- 规则：销售订单自动生成的采购订单允许“付款金额为 0”并继续提交。
- 说明：
  - 本规则优先于规则14，当前版本暂定“无条件放行”，不受保证金放行阈值限制。
  - 后续允许财务手工补录付款单。
  - 手工补录付款单必须关联采购合同，并绑定对应采购订单。

### 12. 仓库出库触发规则
- 状态：`生效`
- 规则：仓库确认实际出库后，系统自动生成出库单。
- 说明：
  - 销售合同对应出库量由两类单据共同构成：仓库正常流程出库单 + 手工补录出库单。
  - 两类出库单均必须绑定销售合同并纳入履约计算。
  - 仓库正常流程出库单必须记录来源回执号或仓库业务单号，并生成唯一幂等键。
  - 双通道履约累计必须防重：`contract_items.qty_out_acc`仅依据“出库单生效防重流水”更新，同一出库单不得重复计入。
  - 手工补录出库单必须携带可追溯业务键（如手工回执号）并参与唯一性校验，避免与系统单重复累计。

### 13. 双阈值配置模型
- 状态：`生效`
- 规则：阈值拆分为两个业务场景独立参数，并采用系统级统一配置下发，不支持合同级人工覆盖。
- 说明：
  - 保证金放行阈值（默认 `1.05`）：用于判断订单是否允许零金额且免凭证。
  - 合同超量履约阈值（默认 `1.05`，即 5%）：用于限制累计实际履约数量上限。
  - 约束：保证金放行阈值不得大于合同超量履约阈值，避免口径冲突。
  - 系统参数变更后，仅影响新生效的业务单据；历史单据按生成时快照口径计算。

### 14. 零金额与免凭证例外
- 状态：`生效`
- 规则：当合同待执行数量（按油品）小于等于保证金对应数量乘以保证金放行阈值时，订单可填写收/付款金额为 0，且可免上传收/付款凭证。
- 说明：
  - 该场景仍需生成收付款单并记录“免凭证原因、操作人、操作时间”。
  - 不适用于规则11所定义的“销售订单衍生采购订单付款=0无条件放行”场景。
  - 本规则仅适用于“订单衍生的普通收款单/付款单”，不适用于合同保证金单据本身。
  - 保证金对应数量的计算口径固定为：`已确认/已核销且未退款的保证金净额 / 当前订单油品对应合同单价`。
  - 待执行数量的计算口径固定为：
    - 销售方向：`contract_item.qty_signed - contract_item.qty_out_acc`
    - 采购方向：`contract_item.qty_signed - contract_item.qty_in_acc`
  - 金额、数量比较时分别沿用规则17和规则20的精度口径：金额 `0.01`，数量 `0.001`。

### 15. 订单定价与实收实付
- 状态：`生效`
- 规则：订单单价来源于对应合同单价，实收实付金额来源于财务确认的收付款单金额。
- 说明：
  - 系统自动生成收付款单，订单金额字段仅用于业务展示与勾稽。
  - 合同附件归属合同；订单业务附件归属订单。
  - 付款凭证附件归属付款单；收款凭证附件归属收款单。
  - 发货指令单及其回传附件归属采购订单。

### 16. 合同数量履约状态
- 状态：`生效`
- 规则：当合同累计实际履约数量达到合同数量时，合同进入“数量履约完成”状态。
- 说明：
  - 数量履约完成不等于合同最终关闭。
  - 合同进入数量履约完成后，禁止新增出库/入库单生效；已创建但未生效单据需转终止或进入手工关闭差异处理。

### 17. 合同金额闭环判定（采购与销售分开）
- 状态：`生效`
- 规则：合同最终关闭前必须满足对应方向的金额闭环公式，且保证金不重复计入。
- 说明：
  - 销售合同：`Σ(各油品累计实际出库数量 × 对应合同单价) = 销售合同累计实收净额`
  - 采购合同：`Σ(各油品累计实际入库数量 × 对应合同单价) = 采购合同累计实付净额`
  - 收款与付款按合同方向独立核算，不得混算。
  - 金额闭环容差固定为 `0.01`（含），即 `|闭环差额| <= 0.01` 视为闭环成立。
  - 保证金“不重复计入”净额口径（固定）：
```text
销售合同累计实收净额
= SUM(普通收款单.amount_actual, status in 已确认/已核销)
+ SUM(保证金收款单.amount_actual, status in 已确认/已核销 and refund_status != 已退款)
- SUM(保证金退款金额.refund_amount, status in 已确认/已核销)

采购合同累计实付净额
= SUM(普通付款单.amount_actual, status in 已确认/已核销)
+ SUM(保证金付款单.amount_actual, status in 已确认/已核销 and refund_status != 已退款)
- SUM(保证金退款金额.refund_amount, status in 已确认/已核销)
```
  - 若当前阶段未启用退款流程，则`refund_status=未退款`、`refund_amount=0`，净额公式仍保持一致。

### 18. 合同最终关闭与手工关闭
- 状态：`生效`
- 规则：合同自动最终关闭需同时满足“数量履约完成 + 金额闭环”；不满足时仅允许授权用户手工关闭。
- 说明：手工关闭必须“原因必填 + 二次确认 + 审计留痕”。

### 19. 手工关闭后的处理顺序
- 状态：`生效`
- 规则：合同手工关闭后，必须按统一顺序处理库存、资金与关联单据边界，防止账实不一致。
- 说明：
  - 第一步：锁定合同与关联单据，禁止新增引用。
  - 第二步：终止未生效单据（入库/出库/收款/付款）并写入终止原因。
  - 第三步：释放未执行库存预占/预留，不回滚已过账库存流水。
  - 第四步：将剩余数量与剩余金额结转为“手工关闭差异记录”，仅用于审计，不再参与后续流转。
  - 第五步：合同与关联单据统一标记“手工关闭”并完成审计归档。

### 20. 合同履约数量上限
- 状态：`生效`
- 规则：合同累计实际履约数量不得超过合同数量乘以合同超量履约阈值（默认 1.05）。
- 说明：
  - 数量计算与校验统一保留到 `0.001` 精度后再比较阈值。
  - 超限时禁止单据生效。

### 21. 单据上下游可追溯
- 状态：`生效`
- 规则：合同、订单、收付款单、出入库单均必须记录上下游关联关系。
- 说明：
  - 关联关系需支持一对多和多对多，满足审计穿透与报表追溯需求。
  - 防重要求：通过`doc_relations`唯一关系约束 + 履约累计防重流水唯一键双重控制，避免双通道重复累计。

### 22. 并发校验策略（当前版本）
- 状态：`生效`
- 规则：当前版本按低并发业务假设运行，不引入分布式锁与高并发强一致控制。
- 说明：若实际并发规模超出当前业务假设，另行立项评估并发架构升级。

### 23. 全新库初始化清单
- 状态：`生效`
- 规则：V6 上线前必须完成主数据与参数初始化清单，未完成不得上线。
- 说明：至少包含：
  - 组织架构、角色权限、审批流模板。
  - 角色与公司绑定基线：客户->客户公司；供应商->供应商公司；仓库->仓库公司；运营/财务/管理员->运营商公司。
  - 客户、供应商、油品、仓库、库位、计量单位。
  - 单据编号规则、合同阈值参数、默认税率与价格精度。
  - 报表维度字典、审计事件编码、附件分类字典。

### 24. 端能力划分总原则
- 状态：`生效`
- 规则：管理后台覆盖全量业务能力；小程序仅承载适合移动端的轻量业务。
- 说明：
  - 系统以运营商公司视角管理全链路，运营、财务、管理员均归属运营商公司。
  - 客户、供应商、仓库角色用户不开放管理后台 Web 登录权限，仅通过小程序处理业务。
  - 任何核心资金结算、主数据维护、复杂审批编排、批量配置操作必须在管理后台完成。

### 25. 小程序功能边界
- 状态：`生效`
- 规则：小程序仅开放移动高频、低复杂度、强时效的业务能力。
- 说明：
  - 建议开放：待办处理、订单发起/查询、出入库执行回执、合同/订单进度跟踪、轻量报表查看。
  - 禁止开放：主数据维护、阈值参数配置、复杂财务对账、批量改单、手工关闭批处理。

### 26. 管理后台看板与仪表盘
- 状态：`生效`
- 规则：管理后台必须提供“经营仪表盘 + 业务看板”两类能力。
- 说明：
  - 仪表盘：经营与履约核心指标（收付款、出入库、合同执行率、异常预警）。
  - 业务看板：待办、阻塞、超阈值风险、流程瓶颈与处理时效。

### 27. 报表分层策略
- 状态：`规划中`
- 规则：小程序与管理后台采用差异化报表策略。
- 说明：
  - 小程序：提供轻量实时汇总报表（查询简单、字段收敛）。
  - 管理后台：提供多维管理报表（合同、订单、油品、客户、仓库、时间、异常等）。

### 28. 报表 SLA 与口径冻结
- 状态：`规划中`
- 规则：报表必须定义时效 SLA 与口径版本冻结机制。
- 说明：
  - 建议小程序核心汇总报表 SLA 为 T+0（5 分钟内更新）。
  - 建议管理后台经营报表 SLA 为 T+0~T+1（按报表级别区分）。
  - 指标口径按版本管理，版本生效后不得在同周期内变更；若变更必须发布“口径变更公告 + 重算范围”。

### 29. 报表生成机制
- 状态：`规划中`
- 规则：采用“事件触发增量更新 + 定时全量校准”的混合机制，不采用纯同步触发式重算。
- 说明：
  - 减少高并发下报表阻塞风险，提升可审计性与可追溯性。
  - 增加合同闭环扫描任务：每日定时扫描“数量履约完成但未关闭”的合同，并推送看板/仪表盘告警。
  - 增加数量履约扫描任务：每日定时扫描“生效中且长时间未更新履约量”的合同，并推送异常提醒。
  - 扫描任务需具备重复触发控制（任务幂等键）并写入审计日志，避免重复告警与重复计算。

### 30. 小程序审批驳回边界
- 状态：`生效`
- 规则：小程序仅允许查看审批驳回结果，不支持执行驳回动作与填写驳回原因。
- 说明：审批驳回动作统一在管理后台完成，保证审计链路集中与操作可控。

### 31. 管理后台仪表盘首版指标冻结
- 状态：`生效`
- 规则：管理后台经营仪表盘首版固定四项核心指标：合同执行率、当日实收实付、库存周转、超阈值告警数。
- 说明：新增或替换首屏核心指标需按“报表口径版本变更”流程执行。

### 32. 非目标：多活与跨地域容灾
- 状态：`弃用`
- 规则：当前版本不建设多活架构与跨地域容灾体系。
- 说明：采用单地域部署与常规备份恢复策略即可满足当前规模。

### 33. 非目标：完整财务核算体系
- 状态：`弃用`
- 规则：当前版本不建设总账、成本核算、税务核算等完整财务系统能力。
- 说明：系统定位为业务链与收付款辅助证据链，不替代专业财务软件。

### 34. 用户可见信息中文化
- 状态：`生效`
- 规则：小程序与管理后台所有用户可见信息必须使用中文显示。
- 说明：
  - 范围包括：系统提示、流程阻断提示、页面字段显示、状态描述、按钮文案、消息通知、报表标题、空态与异常态提示。
  - API错误码可保留英文编码，但返回给用户直接展示的`message`字段必须为中文。
  - 第三方系统回传原文若为非中文，需在展示前完成中文映射，不得直接透传给终端用户。

### 35. 管理后台登录边界与后端接口授权分层
- 状态：`生效`
- 规则：管理后台 Web 登录权限与后端接口授权是两层独立控制，必须同时成立。
- 说明：
  - “管理后台”默认指 `admin-web` 管理页面登录权限；客户、供应商、仓库角色不得登录管理后台 Web。
  - 后端接口权限不得仅依赖前端登录页、菜单隐藏或端类型限制，必须在服务端按“已认证身份 + 角色 + 公司范围 + 接口能力”校验。
  - 小程序接口由小程序已登录身份访问，管理后台接口由管理后台已登录身份访问；服务端必须拒绝越权调用。
  - 系统级参数接口（如阈值配置）属于管理员能力，需在后端接口层校验管理员权限后方可访问。

### 36. 合同提交审批与驳回回退
- 状态：`生效`
- 规则：采购合同、销售合同在进入生效链路前，必须遵循 `草稿 -> 待审批 -> 生效中` 的早期状态流转；待审批被驳回后回退到 `草稿`。
- 说明：
  - 合同录入完成后，先以 `草稿` 保存，再由管理后台执行“提交审批”进入 `待审批`。
  - 合同审批接口仅允许处理 `待审批` 状态，不允许跳过提交直接生效。
  - 审批驳回必须填写原因并写入审计；驳回回退后允许修改合同内容并再次提交审批。

### 37. 销售订单提交审批与驳回回退
- 状态：`生效`
- 规则：销售订单必须遵循 `草稿 -> 待运营审批 -> 待财务审批 -> （驳回 或 已衍生采购订单）` 的审批主链；不同审批节点的驳回回退口径必须固定。
- 说明：
  - 销售订单录入完成后，先以 `草稿` 保存，再提交到 `待运营审批`。
  - 运营审批通过后进入 `待财务审批`；运营驳回回退到 `草稿`。
  - 财务审批通过后，系统进入采购订单与收付款单衍生链路，订单状态更新为 `已衍生采购订单`。
  - 财务驳回时订单状态更新为 `驳回`；驳回态允许修改订单核心字段，且保存修改后必须自动回到 `草稿`，再重新提交审批。

### 38. 销售订单财务审批的采购合同绑定
- 状态：`生效`
- 规则：销售订单财务审批通过时，必须选择一个已生效的采购合同；系统自动生成的采购订单必须默认绑定该采购合同。
- 说明：
  - 被选择的采购合同必须为 `生效中` 的采购方向合同，且油品明细必须覆盖当前销售订单油品。
  - 销售订单衍生采购订单的 `付款金额=0` 例外仅影响付款校验放行，不得豁免采购合同绑定。
  - 财务审批通过后，系统自动生成采购订单实体，并为付款单、收款单写入待处理任务，供后续模块消费生成单据。

### 39. 出库单执行前置状态门禁
- 状态：`生效`
- 规则：仓库正常流程出库单与手工补录出库单，均只能基于已进入执行前置状态的销售订单生成。
- 说明：
  - 允许生成出库单的销售订单状态仅限：`已衍生采购订单`、`执行中`。
  - `草稿`、`待运营审批`、`待财务审批`、`驳回` 等未进入执行阶段的销售订单，禁止生成任何出库单。
  - 该门禁同时适用于仓库正常流程回执与手工补录出库，避免绕过订单审批主链直接形成履约数量。

### 40. 合同关闭收口规则
- 状态：`生效`
- 规则：合同进入自动关闭或手工关闭时，所有未终态且不再参与后续流转的收付款单、入库单、出库单必须统一收口，禁止继续被引用或继续流转。
- 说明：
  - 自动关闭时：剩余的 `草稿`、`待补录金额`、`待提交`、`校验失败` 单据统一转 `已终止`，并写入审计。
  - 手工关闭时：按相同口径终止未完成单据，并额外记录关闭差异金额、关闭差异数量明细与关闭原因。
  - 合同关闭后，手工补录收付款单、补录出库单、出入库再次提交、0金额再次确认等后续流转动作必须被服务端阻断。

### 41. 仪表盘首版指标口径冻结
- 状态：`生效`
- 规则：管理后台仪表盘首版四项核心指标必须按固定公式计算，不允许前后端各自解释。
- 说明：
  - 合同执行率：`Σmin(合同已履约数量, 合同签约数量) / Σ合同签约数量`；采购方向取 `qty_in_acc`，销售方向取 `qty_out_acc`；统计范围仅包含已进入执行阶段的合同，即 `生效中`、`数量履约完成`、`已关闭`、`手工关闭`、`已归档`。
  - 当日实收实付：按 `Asia/Shanghai` 自然日统计 `已确认/已核销` 的收款净额与付款净额，单卡片展示“实收 + 实付”两项值。
  - 库存周转：`近30个自然日已过账出库总量 / 当前可用库存量`；当前可用库存量口径为 `累计已过账入库量 - 累计已过账出库量`，分母小于等于0时返回0。
  - 超阈值告警数：`校验失败` 的入库单与出库单 + `待补录金额` 的收付款单 + `数量履约完成但未关闭` 的合同数量之和。
  - 首版统一使用指标版本 `v1`，后续变更必须走报表口径版本流程。

### 42. 供应商移动端首批真实业务页边界
- 状态：`生效`
- 规则：供应商移动端首批真实业务页仅开放采购订单进度查看、发货准备信息回看、首批附件回传与单笔发货确认，不开放付款确认、异常关闭和后台审批动作。
- 说明：
  - 首批页面目标是补齐供应商角色“可查看、可回看准备信息、可回传首批业务附件、可提交单笔发货确认”的最小闭环，不把复杂资金校验与异常处置搬到小程序。
  - 供应商侧首批查询对象固定为“当前供应商公司名下采购订单”，不得跨公司读取其他供应商订单。
  - 若后端已完成供应商侧采购订单列表/详情接口冻结，而附件接口尚未冻结，前端仅允许落真实进度页与附件入口占位，不得伪造上传结果、已回传状态或附件明细。
  - 付款校验、手工关闭、异常驳回、批量处理仍保留在管理后台，由运营/财务按审计链路处理。

### 43. 供应商附件回传首批开放边界
- 状态：`生效`
- 规则：供应商附件回传首批仅开放“盖章发货指令单/回单类附件”的上传与回看，不开放付款凭证确认、不开放异常关闭、不开放跨订单批量上传。
- 说明：
  - 该批次目标是补齐供应商协同中的审计留痕闭环，让供应商在采购订单详情页可上传业务附件，并让运营侧可追溯查看。
  - 附件归属必须绑定到具体采购订单或其下游单据，不允许出现无法追溯业务来源的裸附件记录。
  - 首批仅允许供应商公司在小程序上传本公司名下采购订单附件；附件类型、数量上限、命名规则和业务标签必须由服务端校验。
  - 真实文件存储能力若未冻结完成，可先走受控文件路径或对象存储占位方案，但必须保留审计字段、上传人、上传时间与业务标签。
  - 测试环境产生的附件路径或测试附件记录，必须使用 `CODEX-TEST-`、`AUTO-TEST-` 等统一可检索标识，并在回归后定向清理。
  - 付款确认仍保留在运营/财务后台，不因附件回传开放而改变资金审批职责边界。

### 44. 供应商发货确认首批开放边界
- 状态：`生效`
- 规则：供应商发货确认首批仅开放对“当前供应商公司名下采购订单”的单笔确认提交，并将采购订单状态从`待供应商确认`推进到`供应商已确认`；不开放付款确认、不开放异常关闭、不开放批量确认。
- 说明：
  - 该批次目标是在附件回传之后补齐供应商侧最小状态推进闭环，让供应商能在移动端明确确认“已完成发货准备并提交确认”。
  - 发货确认必须以采购订单归属校验、当前状态校验、确认时间留痕和操作人留痕为前提，不允许跳过状态直接推进到`待付款校验`或后续状态。
  - 若采购订单尚无首批业务附件，也不得强制阻断确认；但页面需明确提示“建议先补齐附件留痕后再提交确认”，避免形成伪强校验。
  - 发货确认提交后，付款校验与后续执行仍由运营/财务/仓储链路继续推进，供应商侧不获得资金审核与异常处置权限。
  - 测试环境产生的确认数据、备注或附件引用，仍需使用统一可检索标识并在回归后定向清理。

### 45. 供应商付款校验结果回看首批开放边界
- 状态：`生效`
- 规则：供应商付款校验结果回看首批仅开放对“当前供应商公司名下采购订单”的只读状态回看，用于展示当前是否进入`待付款校验`、`可继续执行`等结果提示；不开放供应商侧付款确认、付款驳回、异常关闭、批量处理与资金凭证审核。
- 说明：
  - 该批次目标是在供应商发货确认之后补齐“确认后发生了什么”的只读反馈，避免供应商只能看到状态名却无法理解是否已进入后续付款校验链路。
  - 首批仅开放结果回看与中文提示，不开放任何会改变资金状态的供应商操作；资金校验、付款确认与例外放行仍由运营/财务后台执行。
  - 页面需区分普通采购订单与`付款金额=0`例外场景，明确提示“待付款校验中”“已通过付款校验，可继续执行”“命中例外放行，等待后续补录”等冻结口径；若订单已进入`执行中`，仅允许展示“已完成付款校验并进入后续执行”的只读承接提示。
  - 回看范围必须绑定到具体采购订单，且仅允许当前供应商公司查看，不得跨公司读取其他供应商订单的付款校验结果或资金说明。
  - 若当前后端尚未形成完整付款校验结果快照，首批可先基于现有订单状态、例外标记与受控说明字段做只读展示，但不得伪造“已付款”或“财务已终审”结论。

### 46. M8后台迁移优先级基线

- 规则：`M8-16` 起后台迁移优先级应优先补齐运营/财务/管理员的合同、订单、资金、库存处理页面与异常处置入口，再继续扩展供应商侧新动作或低优先级展示页。

  - 供应商、仓库、客户等协同角色后续新增能力，不得挤占运营/财务关键后台闭环的优先级，除非存在明确监管、上线阻断或安全合规要求。
  - 路线文档、任务包与 README 中对“已实现/未实现/规划中”的口径必须一致，避免继续出现供应商文档连续细拆而后台能力仅在大包中笼统描述的情况。

### 48. 财务资金处理台首批开放边界
- 状态：`规划中`
- 规则：财务资金处理台首批仅开放收款单/付款单列表筛选、详情回看、手工补录、单笔确认与凭证路径回看；不开放退款/核销、不开放驳回待审核独立处理台、不开放批量处理与真实文件上传。

## 业务目标与场景（阶段基线）
- 业务目标：
  - 在不超过50用户的规模下，完成合同、订单、收付款、出入库的主链路协同管理。
  - 为财务提供可穿透、可追溯的辅助证据链，不替代专业财务核算系统。
  - 以运营商视角统一管理客户、供应商、仓库协同执行与异常处置。
- 核心场景：
  - 合同驱动订单执行与资金单据自动生成。
  - 仓库实际出入库驱动合同履约数量更新。
  - 关键节点凭证与附件留痕，支持审计穿透到上下游单据。
  - 仪表盘与看板对经营异常、超阈值风险进行可视化预警。
