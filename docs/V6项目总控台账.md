# V6 项目总控台账

## 1. 文档定位
- 文档状态：`生效`
- 目标：以项目总控视角固定 V6 的宏观阶段、当前唯一焦点、阶段门禁、纠偏结论与遗留项收口规则，避免后续开发围绕局部功能扩散并再次误判为“可上线”。

## 3. 当前项目快照
- 当前日期：`2026-03-08`
- 当前所在阶段：`阶段 C - 生产可用版本开发重启准备完成`
- 当前完成边界：旧版 `M1 ~ M7`、`M8-01 ~ M8-28` 与阶段 D 首轮联调结果，仅证明旧版交付范围达到“可联调”；本轮已完成对“生产可用版本”目标的宏观纠偏，新增治理总纲、阶段 E 清单与追踪矩阵，后续开发不再以旧版联调结论替代生产化门禁。
- 当前唯一主目标：`G1 前置治理基座开发`
- 当前非目标：
  - 不再把 `demo/proxy/local_api/mock/dev-login/debug_openid` 当作正式交付能力；
  - 不在 `G1` 未完成前并行开启 `G2 ~ G5` 的核心开发；
  - 不以“旧版联调已通过”为由跳过正式身份、治理基座与订单后半状态机收口。
- 下一固定路径：`G1 前置治理基座开发 -> G2 正式身份与端收口 -> G3 订单后半状态机闭环 -> G4 真实业务全链路回归 -> G5 发布与上线准备 -> 阶段 D -> 阶段 E`。

## 6. 当前唯一焦点与待评估池
- 当前唯一焦点：`G1 前置治理基座开发`。
- 顺序排队项：`G2 正式身份与端收口`、`G3 订单后半状态机闭环`、`G4 真实业务全链路回归`、`G5 发布与上线准备`。
- 待评估池：`阶段 E 治理增强池（ADM-TRACE-01、ADM-AUDIT-01） + 本轮正式开发中新增的非阻断优化项`。
- 准入规则：待评估池事项不得直接插入当前开发，必须在当前唯一焦点通过模块门禁后，再进入下一轮冻结。

## 7. 变更控制与收口规则
- `唯一入口`：所有新增事项先进入待评估池，必须标明属于业务规则、页面补齐、缺陷修复、上线阻断还是技术债收口。
- `唯一焦点`：同一时间仅允许一个主目标批次处于执行中；其他事项只能排队，不并行扩散。
- `唯一出口`：阶段 A/B/C/D/E 只能顺序流转；若发现阶段判断错误，必须先回退总控状态并冻结唯一纠偏目标。
- `规则变更前置`：凡涉及业务规则新增、调整、停用，必须先更新 `docs/需求方案.md` 并标记状态，再进入开发。
- `正式交付边界`：`demo/proxy/local_api/mock/dev-login/debug_openid` 仅属于开发工具能力，不得作为真实业务测试准入、验收通过或生产运行依据。
- `遗留项唯一去向`：任何降级、延后、留坑事项，必须同步写入 `docs/V6生产可用版本治理总纲.md`、`docs/V6阶段E-验收发布与上线清单.md` 或 `docs/V6规则-实现-测试-验收追踪矩阵.md`，不得口头漂移。
- `跨模块约束`：若新增事项跨越多个核心模块，必须先完成上下游边界定义与评审，未冻结不得直接开工。
