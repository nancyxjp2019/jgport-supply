## 3. 各单据关闭逻辑

### 3.1 销售订单（`SalesOrderV5`）
- 关闭入口1：`PATCH /api/v1/sales-orders/{id}/reject`。
- 执行角色：`ADMIN`。
- 可关闭前置状态：`SUBMITTED`、`OPERATOR_APPROVED`。
- 关闭结果：状态置为 `REJECTED`，写入 `closed_reason`、`closed_by`、`closed_at`；如存在冻结库存则释放。
- 联动：仅释放本单冻结库存，不强制联动采购订单。

- 关闭入口2：`PATCH /api/v1/sales-orders/{id}/abnormal-close`。
- 执行角色：`ADMIN`。
- 可关闭前置状态：`CUSTOMER_PAYMENT_CONFIRMED`、`READY_FOR_OUTBOUND`。
- 关闭结果：状态置为 `ABNORMAL_CLOSED`，写入关闭原因与关闭人信息；释放冻结库存。
- 联动：若已有关联采购订单且采购订单未终态，则采购订单同步置为 `ABNORMAL_CLOSED`。

- 自动终态：采购订单“仓库出库完成”后，销售订单同步置为 `COMPLETED`（该路径不写 `closed_reason` 字段）。

- 代码定位：
  - `backend/app/api/routers/v5_sales_orders.py:218`
  - `backend/app/api/routers/v5_sales_orders.py:252`
  - `backend/app/services/v5_sales_order_service.py:689`
  - `backend/app/services/v5_sales_order_service.py:713`
  - `backend/app/services/v5_sales_order_service.py:406`
  - `backend/app/services/v5_sales_order_service.py:360`
  - `backend/app/services/v5_purchase_order_service.py:443`

### 3.2 采购订单（`PurchaseOrderV5`）
- 关闭入口：`PATCH /api/v1/purchase-orders/{id}/abnormal-close`。
- 执行角色：`ADMIN`。
- 可关闭前置状态：`PENDING_SUBMIT`、`SUPPLIER_PAYMENT_PENDING`、`SUPPLIER_REVIEW_PENDING`、`WAREHOUSE_PENDING`。
- 关闭结果：状态置为 `ABNORMAL_CLOSED`，写入 `closed_reason`、`closed_by`、`closed_at`。
- 联动：
  - 释放关联销售订单冻结库存；
  - 关联销售订单同步置为 `ABNORMAL_CLOSED`。

- 被动关闭：销售订单执行异常关闭时，也会反向把关联采购订单置为 `ABNORMAL_CLOSED`（若采购单尚未终态）。

- 自动终态：`PATCH /api/v1/purchase-orders/{id}/warehouse-outbound` 成功后，采购订单置为 `COMPLETED`，并同步销售订单 `COMPLETED`。

- 代码定位：
  - `backend/app/api/routers/v5_purchase_orders.py:195`
  - `backend/app/services/v5_purchase_order_service.py:578`
  - `backend/app/services/v5_purchase_order_service.py:274`
  - `backend/app/services/v5_purchase_order_service.py:443`
  - `backend/app/services/v5_sales_order_service.py:723`

- 当前实现差异提醒：`SUPPLIER_PAYMENT_PENDING` 仍在状态枚举与关闭校验中，但现行“提交采购订单”流程会直接流转到 `SUPPLIER_REVIEW_PENDING`。
  - 代码定位：`backend/app/services/v5_purchase_order_service.py:389`

### 3.3 销售合同（`SalesContract`）
- 手工关闭入口：`POST /api/v1/sales-contracts/{id}/void`。
- 执行角色：`FINANCE`、`OPERATOR`、`ADMIN`。
- 可关闭前置状态：仅 `DRAFT`、`PENDING_EFFECTIVE`。
- 约束：若已关联合同订单（销售订单）则禁止作废。
- 关闭结果：状态置为 `VOIDED`，写入 `voided_at`、`voided_by`。

- 自动终态：仓库出库驱动合同执行量累计，当 `pending_execution_qty <= 0` 时自动置为 `COMPLETED`。

- 代码定位：
  - `backend/app/api/routers/v5_contracts.py:206`
  - `backend/app/services/v5_contract_service.py:47`
  - `backend/app/services/v5_contract_service.py:832`

  - `backend/app/services/v5_purchase_order_service.py:230`

### 3.4 采购合同（`PurchaseContract`）
- 手工关闭入口：`POST /api/v1/purchase-contracts/{id}/void`。
- 执行角色：`FINANCE`、`OPERATOR`、`ADMIN`。
- 可关闭前置状态：仅 `DRAFT`、`PENDING_EFFECTIVE`。
- 约束：若已关联采购订单或采购入库单则禁止作废。
- 关闭结果：状态置为 `VOIDED`，写入 `voided_at`、`voided_by`。

- 自动终态：与销售合同一致，仓库出库驱动执行量累计，`pending_execution_qty <= 0` 自动置为 `COMPLETED`。

## 4. 业务执行与风险评估（按当前实现）
- 业务可执行性：双订单与双合同的终态流转闭环已经可执行，终止动作集中在 `ADMIN`，职责边界清晰。
- 运营效率与成本：采购单提交已合并付款凭证上传并直达供应商审核，减少一步人工流转，但保留了未使用状态分支，后续需统一。
- 运输与仓储安全风险：异常关闭与驳回均释放冻结库存，出库完成路径校验库存与出库上限（110%），能降低库存穿透风险。
- 合规与可追溯：关闭/终态动作均写业务日志，且订单、合同均保留关闭人/时间字段，满足审计追踪。
- 实施复杂度与迭代优先级：优先建议补齐“采购入库单作废”闭环，或显式弃用 `SUPPLIER_PAYMENT_PENDING`，避免状态定义与实际流程长期偏离。
