# 隽港顺达供应链项目 环境迁移手册

## 1. 文档信息
- 版本：V5
- 更新日期：2026-03-05
- 适用阶段：V5 开发完成后，测试/生产环境迁移与部署

## 2. 目标
- 保证代码在不同环境可复现运行，避免“本地可用、服务器不可用”。
- 保证 V5 环境切换、数据库结构升级、初始化数据准备、配置变更与回滚过程可追溯、可恢复。
- 保证安全配置（密钥、证书、权限）在迁移过程中不泄漏、不丢失。

## 6. 发布前环境固化（开发机执行）

### 6.1 导出 Conda 历史依赖（推荐）
```bash
mkdir -p deploy
conda env export -n jgport --from-history > deploy/environment-history.yml
```

### 6.2 导出 Conda 精确依赖（用于严格复现）
```bash
conda list -n jgport --explicit > deploy/environment-explicit.txt
```

### 6.3 导出 Pip 锁定依赖（备用）
```bash
conda run -n jgport python -m pip list --format=freeze > deploy/requirements-lock.txt
```

## 7. 服务器迁移步骤（标准流程）
1. 初始化 Ubuntu 24.04 LTS 服务器，创建部署用户与目录。
2. 安装 Miniconda（或使用系统 Python + venv 方案）。
3. 拉取仓库代码到目标目录。
4. 依据 `deploy/environment-history.yml` 创建环境。
5. 注入环境变量（仅使用服务器本地密钥文件，不进入 Git）。
6. 配置 PostgreSQL 账号权限、连接参数、备份目录与附件目录权限。
7. 配置日志目录（如 `backend/logs/`）并确认进程用户可写。
8. 启动应用并执行数据库迁移。
9. 验证系统日志与业务日志是否正常写入。
10. 配置 Nginx HTTPS 反代并启用访问日志。
11. 执行冒烟验证（下单、收款确认、发货、完结、看板查询）。
12. 发布成功后做一次完整备份并记录版本号。

## 8. PostgreSQL 备份与恢复要求

### 8.1 切换前备份
- 使用 PostgreSQL 逻辑备份（`pg_dump`）与附件元数据同步备份。
- 生产库建议使用自定义格式（`-Fc`）便于按对象恢复。

```bash
export PGPASSWORD='请替换为数据库密码'
pg_dump -h 127.0.0.1 -p 5432 -U jgport -d jgport_prod -Fc -f /backup/jgport_$(date +%F_%H%M%S).dump
```

### 8.2 关键配置
- 建议启用定时逻辑备份：每日全量备份，默认保留最近 7 份。
- 建议同时启用 WAL 归档，支持时间点恢复（PITR）。
- 恢复命令示例（空库恢复）：

```bash
pg_restore -h 127.0.0.1 -p 5432 -U jgport -d jgport_prod --clean --if-exists /backup/jgport_2026-03-05_230000.dump
```

## 9. 安全迁移检查项（强制）
- `.env` 不入库，仓库只保留 `.env.example`。
- JWT 密钥、对象存储密钥、数据库账号权限最小化。
- Nginx 强制 HTTPS，禁用明文管理入口。
- 敏感信息（身份证、手机号）默认脱敏显示。
- 上传附件限制为图片/PDF，限制文件大小并记录审计日志。

## 10. 迁移后验收清单
- API 健康检查通过。
- 销售订单状态机流转正常（`SUBMITTED -> OPERATOR_APPROVED -> CUSTOMER_PAYMENT_CONFIRMED -> READY_FOR_OUTBOUND -> COMPLETED`）。
- 采购订单状态机流转正常（`PENDING_SUBMIT -> SUPPLIER_PAYMENT_PENDING -> SUPPLIER_REVIEW_PENDING -> WAREHOUSE_PENDING -> COMPLETED`）。
- 库存预占、实扣、释放逻辑与文档一致。
- 仓库上传出库单后可完成订单。
- 运营可查看订单/金额/库存看板。
- 仓库与客户只能访问各自权限范围内数据。
- 审计日志可追溯关键操作（收款确认、代完结、异常关闭）。

## 11. 回滚方案（最小可行）
1. 停止新版本服务。
2. 恢复上一个可用版本代码。
3. 恢复最近一次完整数据库备份。
4. 校验关键订单与库存数据一致性。
5. 重新启动旧版本并执行健康检查。
6. 记录回滚原因与影响范围，形成复盘条目。
